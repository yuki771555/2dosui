# **1\. 背景 — こういう状況・社会になってきた**

## **1.1 睡眠への関心が急速に高まっている**

近年、生産性・健康・メンタルウェルネスへの社会的関心の高まりとともに、「睡眠の質」は個人の重要な管理指標となった。スマートウォッチやスマートフォンアプリによる睡眠ログの記録は一般化し、「何時間寝たか」「深い眠りが何分あったか」を日常的に把握する人が増えている。

また、リモートワークの普及により在宅勤務者が増えたことで、起床時刻の自己管理の重要性はさらに増している。オフィスへの通勤という物理的強制力がなくなった環境では、自分自身で起床リズムを維持することが生活の質と仕事のパフォーマンスに直結する。

## **1.2 IoT・センシング技術の民主化**

Raspberry PiやArduino、安価なセンサーモジュールの普及により、専門的なハードウェア知識がなくても、自宅でセンサーデバイスを構築できる時代になった。ロードセル+HX711の組み合わせによる荷重計測は、体重計のDIY用途として広く知られており、関連するライブラリや情報も豊富に整備されている。

こうした民主化されたIoT技術は、生活課題を自分でハックして解決する「個人DIYデバイス」の可能性を大きく広げている。市販品では実現できない、自分の生活に最適化されたシステムを低コストで作ることが、今や現実的な選択肢となっている。

## **1.3 「睡眠の計測」から「行動の検知」へ**

従来の睡眠デバイスは、睡眠中の状態(深い眠り・浅い眠りなど)を記録・可視化するものが中心だった。しかし、真に問題なのは「睡眠の質」だけでなく、「起きるべき時間に確実に起きられるか」という起床行動そのものである。計測から一歩踏み込んだ「行動の検知と介入」へのニーズが、個人レベルでも高まってきている。

|  | 時代背景のまとめ 睡眠管理への関心の高まり × IoT技術の民主化 × 在宅勤務による自己管理の必要性 — この3つが重なり、「自分だけの起床支援デバイスをDIYで作る」ことの意義と実現可能性が、今まさに高い水準で揃っている。 |

# **2\. 問題 — でも、こういう問題がある**

## **2.1 アラームは「鳴る」が「起こす」ことはできない**

スマートフォンや目覚まし時計のアラームは、指定した時刻に音を鳴らすことはできる。しかし「ベッドから出たかどうか」「出た後に戻っていないか」を検知する機能は持っていない。アラームを止めた後の行動は、完全にユーザー自身の意志力に委ねられている。

## **2.2 2度寝は「意図的な行動」ではない**

多くの場合、2度寝は「もう少し寝よう」という明確な意思決定ではなく、半覚醒状態でベッドに戻り、気づいたら寝ていた、という非意図的な行動である。このため、意志力や動機付けだけでは解決しにくく、外部からのフィードバックや介入が効果的とされている。

## **2.3 既存の解決手段の限界**

| 手段 | できること | できないこと・限界 |
| ----- | ----- | ----- |
| **音アラーム(スマホ等)** | 時刻に音を鳴らす | 再入床の検知ができない。止めた後は無力 |
| **スヌーズ機能** | 複数回鳴らす | 慣れると無意識にOFFにするだけになる |
| **光目覚まし** | 自然な光で覚醒を促す | 起きた後の行動を一切追跡できない |
| **スマートウォッチ** | 睡眠状態の記録・分析 | 装着必須。再入床の検知は行わない |
| **睡眠追跡アプリ** | 睡眠サイクルの記録 | 「起きた後に戻った」を物理的に検知できない |

## **2.4 問題の本質**

|  | 検知できていないから介入できない 既存ツールはすべて「起きる前」に作用するもの(アラーム)か「起きた後の記録」にとどまるもの(ログ系)である。「一度起きたのに再びベッドに戻った」という行動を、リアルタイムで物理的に検知して介入するツールが存在しない。これが解決されていない本質的な問題である。 |
| :---- | :---- |

# **3\. 提案 — そこで、こういう方法・デバイスを提案する**

## **3.1 提案の概要**

ベッドの4本脚の下にロードセル(荷重センサー)を設置し、Raspberry Pi 4 で荷重変化をリアルタイムに監視する。アラーム後に離床を検知し、その後30分を「監視ウィンドウ」として再入床を監視。再び荷重が増加し数分継続した場合に「2度寝」と判定し、ブザーやスマホ通知で強く介入する。

|  | 提案の核心 音ではなく「荷重の物理変化」を使う。装着不要で、ベッドそのものがセンサーになる。「起きた後に戻ったかどうか」をベッドが直接検知し、自動で介入する。 |
| :---- | :---- |

## **3.2 システム全体構成**

4つのブロックで構成される。圧力変化はアナログ信号として取得し、HX711 で24ビット精度のデジタル信号に変換してから Raspberry Pi 4 で処理する。

  \[ロードセル ×4\]  ──\>  \[HX711\]  ──\>  \[Raspberry Pi 4\]  ──\>  \[ブザー / 通知\]  
                                                                             
   アナログ信号      デジタル(GPIO)     判定・状態遷移      アラート出力  
  (荷重の変化)     (24bit ADC増幅)    (Python / 状態遷移)  (ブザー+Wi-Fi)

## **3.3 判定のしくみ — 状態遷移モデル**

単純な「荷重があるか」ではなく、起床から再入床までの流れを5つの状態で追跡する。

| \# | 状態 | 日本語 | 遷移条件 |
| :---: | ----- | ----- | ----- |
| **1** | **SLEEPING** | 睡眠中 | 初期状態。ベッドに荷重が安定してある |
| **2** | **LEFT\_BED** | 離床 | 荷重が大きく減少 → 起床と判定。監視開始 |
| **3** | **AWAKE\_WINDOW** | 監視中 | 起床後30分間の監視ウィンドウ |
| **4** | **RETURNED** | 再入床 | 監視中に荷重が再び増加 |
| **5** | **2ND\_SLEEP** | 2度寝確定 | 再入床状態が3〜5分以上継続 → 通知発動 |

## **3.5 実装計画 — 段階的アプローチ**

いきなり完成版を目指すのではなく、試作1号(センサー1個・椅子で検証)で動作を確認してから本番構成(ベッド4個)へ拡張する。学校環境でも安全に研究でき、問題の切り分けが容易になる。

### **Phase 1: 試作1号 — 椅子を用いたセンサー1個での動作確認**

本研究は学校での研究活動として実施するため、試作1号は椅子を使って行う。椅子の脚の下に1個のロードセルを設置し、Raspberry Pi 4 と HX711 で荷重変化を取得する。「座った/立った/戻った」という動作がベッドの「乗った/降りた/戻った」と同じ原理で検知できるため、学校のどこでも安全に実験・デモが可能である。

|  | 椅子で試作する理由 ①学校環境でも設置・実験が簡単にできる。②ベッドと同じ「座る/立つ/戻る」の荷重変化パターンを再現できる。③配線・HX711動作・ノイズの問題を切り分けるのに十分な環境。④1個構成で確認後、ベッド4個構成へ知識をそのまま転用できる。 |
| :---- | :---- |

| 確認項目 | 合格基準 |
| ----- | ----- |
| **Raspberry Pi 4 \+ HX711から値が読める** | Pythonコードで荷重値が毎秒出力される |
| **座った/立ったが検知できる** | 値が明確に増減し、着座/離席に追従する |
| **キャリブレーションが通る** | 体重が±数kg以内で表示される |
| **再着席の検知ができる** | 離席後に戻った際、荷重変化として捉えられる |
| **CSVログが保存される** | 授業1コマ分(90分)のログが途切れず保存される |

### **Phase 2: 本番構成 — ベッド4個で精度向上**

* 4個のハーフブリッジをフルブリッジに合成して接続

* キャリブレーションを再実施(総荷重が変わるため)

* 状態遷移コードと2度寝判定ロジックを実装

* ブザーとWi-Fi通知(Discord/LINE)を接続

1〜2週間のログを収集後、ラベル付けして軽量分類モデル(ロジスティック回帰・ランダムフォレスト等)を学習。誤検知をさらに減らし、個人の行動パターンに最適化する。詳細は第5章参照。

## **3.6 延長配線の実作業 — はんだ付けによる1m延長**

ロードセルの付属ケーブルは約20〜25cmしかなく、ベッド中央のHX711まで届かない。普通の導線(ビニール被覆付き)を購入し、はんだ付けで延長する。

### **材料**

* 導線: ビニール被覆付き AWG28〜24、1m×4本(赤・黒・白・緑の4色推奨)

* はんだごて: 20〜40W(温調機能付きが理想)

* はんだ: 電子工作用 φ0.8〜1.0mm

* 熱収縮チューブ: 内径2〜3mm(絶縁テープで代用可)

* ワイヤーストリッパー(カッター代用可)

### **手順(1本の線を延長、4本繰り返す)**

5. ロードセルの既存線の先端1cmの被覆を剥く

6. 延長導線も先端1cmを剥く

7. 先に熱収縮チューブを延長導線に通しておく(後では通せない)

8. 2本の芯線を重ねて指でしっかりねじり合わせる

9. はんだごてで温め、はんだを流し込む(線が温まってからはんだを当てる)

10. 熱収縮チューブを接続部分に移動させ、ライターで収縮・絶縁する

11. 残り3本でも同じ手順を繰り返す

### **ノイズ対策: ツイスト処理**

  \[赤\] \+ \[黒\] → 2本をペアにして軽くねじる  
  \[白\] \+ \[緑\] → 2本をペアにして軽くねじる  
  さらに2ペアをねじり合わせるとより効果的(任意)

|  | はんだ付けのコツ 「はんだごてを当てる → 線が温まる → はんだを当てる」の順番が正しい。いきなりはんだをこてに付けても流れない。芯線が十分温まってからはんだを当てるとスムーズに染み込む。 |
| :---- | :---- |

## **3.7 キャリブレーション手順**

ロードセルの生の値を実際の重さ(kg)に変換するための必須作業。飛ばすと判定ロジックが動作しない。

| Step | 作業 | 内容 | ポイント |

## **5.1 AIによる個人最適化 — ルールベースの先へ**

Phase 1〜4のルールベース実装が安定稼働し始めたら、蓄積されたログデータを使って機械学習モデルを構築することができる。これにより、固定の閾値では対応しきれない「個人差」や「状況の違い」を吸収し、2度寝判定の精度を大幅に向上させることができる。

### **なぜAIが必要か — ルールベースの限界**

ルールベース判定は「荷重が閾値を超えてX分継続したら2度寝」という固定条件で動く。これは汎用的だが、以下のような状況で誤検知・見逃しが発生しやすい。

| 状況 | ルールベースの反応 | 本来の正解 |
| ----- | ----- | ----- |
| 布団の上に荷物を置いた | 2度寝と誤判定することがある | 2度寝ではない |
| 体が小さく荷重が軽い | 閾値を超えずに見逃す場合がある | 再入床として検知すべき |
| ベッドの端に座っただけ | 横になったと誤判定する場合がある | 2度寝ではない |
| 休日の朝は条件を変えたい | 固定条件のため対応できない | 曜日・状況に応じて柔軟に判定したい |

### **AIで使う特徴量(入力データ)**

ロードセル4個から毎秒記録されるログには以下の豊富な特徴量が含まれる。これらをモデルの入力として使う。

| 特徴量 | 説明と意味 |
| ----- | ----- |
| **総荷重** | 4個の合計値。「重い \= 人が横になっている」の基本指標 |
| **荷重の偏り(左右・前後)** | 4点の差分。横になると均等、腰掛けると前側に偏る。姿勢の区別に有効 |
| **荷重変化の速さ** | 急増=飛び乗った、緩増=ゆっくり横になった。行動の種類を区別できる |
| **継続時間** | 一定以上の荷重が何分続いているか。短ければ腰掛け、長ければ就寝の可能性 |
| **時刻** | アラーム後7時10分か8時かで文脈が違う。同じ荷重でも意味が変わる |
| **アラームからの経過時間** | 起床直後(5分以内)の再入床は危険度が高い。時間が経つほどリスク低下 |
| **過去7日間のパターン** | 「この人は月曜日に2度寝しやすい」などの個人クセを学習できる |

### **ラベル付け — 教師データの作り方**

機械学習には「正解データ(教師データ)」が必要。1〜2週間のログを実際の行動と照らし合わせて以下の4クラスのラベルを付ける。

* sleeping: ベッドで睡眠中(安定した荷重)

* sitting: 端に腰掛けた(荷重が短時間・偏りあり)

* left\_bed: ベッドから離れた(荷重の急減)

* second\_sleep: 2度寝(再入床後に荷重が数分継続)

ラベル付けは手動で行う。CSVログを見ながら「この時間帯は腰掛けただけ」などを記録する。数時間分から始めても十分なデータになる。

### **使用するモデルとその特徴**

Raspberry Pi 4 のスペックでリアルタイム推論が動くよう、軽量なモデルを使う。

| モデル | 特徴 | 本システムでの役割 |
| ----- | ----- | ----- |
| **ロジスティック回帰** | 非常に軽量・解釈しやすい | まず最初に試す。特徴量の重要度が数字で見える |
| **決定木** | ルールが可視化できる | 「なぜ2度寝と判定したか」を人間が確認・説明できる |
| **ランダムフォレスト** | 精度が高い・過学習しにくい | ロジスティック回帰より精度を上げたい段階で使う |
| **XGBoost / LightGBM** | 精度が非常に高い | データが十分蓄積された後の最終精度向上に使う |

### **学習〜推論の流れ(Pythonコード例)**

\# step1\_features.py — 特徴量を作る  
import pandas as pd  
df \= pd.read\_csv('bed\_log.csv')

df\['total\_weight'\]   \= df\[\['lc1','lc2','lc3','lc4'\]\].sum(axis=1)  
df\['diff\_lr'\] \= (df\['lc1'\]+df\['lc2'\]) \- (df\['lc3'\]+df\['lc4'\])  \# 左右差  
df\['diff\_fb'\] \= (df\['lc1'\]+df\['lc3'\]) \- (df\['lc2'\]+df\['lc4'\])  \# 前後差  
df\['delta'\]   \= df\['total\_weight'\].diff().abs()                  \# 変化量  
df\['hour'\]    \= pd.to\_datetime(df\['timestamp'\]).dt.hour

\# step2\_train.py — モデルを学習  
from sklearn.ensemble import RandomForestClassifier  
from sklearn.model\_selection import cross\_val\_score  
import joblib

labels \= pd.read\_csv('labels.csv')  
X \= df\[\['total\_weight','diff\_lr','diff\_fb','delta','hour'\]\]  
y \= labels\['label'\]

model \= RandomForestClassifier(n\_estimators=100, random\_state=42)  
scores \= cross\_val\_score(model, X, y, cv=5)  \# 5分割交差検証  
print(f'精度: {scores.mean():.2f} (±{scores.std():.2f})')

model.fit(X, y)  
joblib.dump(model, 'sleep\_model.pkl')

\# step3\_predict.py — Pi 4でリアルタイム推論  
model \= joblib.load('sleep\_model.pkl')  
features \= \[total\_w, diff\_lr, diff\_fb, delta, hour\]  
label \= model.predict(\[features\])\[0\]  
\# → 'second\_sleep' なら通知発動

### **精度評価の指標**

「正確さ(Accuracy)」だけでなく、誤検知(False Positive)と見逃し(False Negative)をバランスよく評価するF1スコアを主指標として使う。

| 指標 | 意味 | 目標値 |
| ----- | ----- | ----- |
| **Accuracy(正確率)** | 全判定のうち正解した割合 | 90%以上 |
| **Precision(適合率)** | 「2度寝」と判定したうち実際に2度寝だった割合 | 誤通知を減らしたい場合に重視 |
| **Recall(再現率)** | 実際の2度寝のうち検知できた割合 | 見逃しを減らしたい場合に重視 |
| **F1スコア** | PrecisionとRecallの調和平均 | 0.85以上を目標(主指標) |
|  | **研究テーマとしてのAI活用** 「ルールベース vs 機械学習モデル vs LLM補助」の3段階比較はそのまま研究の評価軸になる。誤検知率・見逃し率・F1スコアをKPIとして計測し、各アプローチの優劣を定量的に示せると、研究発表として非常に説得力がある。 |  |

## **5.2 LLMの活用 — MLと組み合わせた高度な判断・対話**

機械学習モデルによる数値分類に加えて、LLM(大規模言語モデル)を組み合わせることで、「なぜそう判定したか」の説明生成や、ログデータの自然言語での解釈・対話が可能になる。実現方法は大きく2つに分かれる。

| LLMの役割 | できること | MLとの違い |
| ----- | ----- | ----- |
| **判定理由の説明生成** | 「7:15に離床後、7:22に再入床し5分継続。2度寝と判定」のような自然文を生成 | MLは0/1の分類のみ。LLMは根拠を言葉にできる |
| **ログの傾向分析** | 「先週の起床ログを分析して傾向を教えて」に自然言語で答える | グラフ不要。会話形式で洞察が得られる |
| **曖昧なケースの補助判断** | MLが迷ったケース(確信度50〜60%)をLLMに文脈つきで問い合わせ | MLは数値しか見ないが、LLMは時刻・曜日・状況の文脈を考慮できる |
| **研究考察の補助** | 実験結果のCSVをLLMに渡して「考察を日本語で書いて」と依頼 | 論文・発表資料の考察文作成を補助 |

## **5.3 センサーの拡張**

圧力センサーに加えて、補助センサーを追加することで判定精度をさらに高められる。

| センサー | 取得できる情報 | 2度寝判定への貢献 |
| ----- | ----- | ----- |
| **人感センサー(PIR)** | 部屋内に人がいるか | 「歩き回っている \= 起きている」として2度寝リスクを下げる |
| **ドア開閉センサー** | 部屋を出たかどうか | 「部屋を出た \= 起床確定」として監視を解除できる |
| **カメラ(Camera Module)** | 姿勢・体の向き | 「横になった姿勢」を視覚的に確認。誤検知を最小化 |
| **マイク(音声検知)** | 声・アラーム音 | 「声を出している \= 起きている」として状態を補正 |

## **5.4 通知・介入の強化**

現在の「ブザー \+ スマホ通知」から、より強力な介入手段へ発展できる。

* スマートライト連携: 2度寝検知時に自動で照明を最大輝度に点灯

* スマートスピーカー連携: 「2度寝を検知しました、起きてください」と音声アナウンス

* 家族・同居人への通知: LINE/Discordで家族のスマホにも通知

* 段階的エスカレーション: 5分後に再確認し、まだ寝ていたら音量を上げる

蓄積されたログから、自分の睡眠・起床行動を分析・可視化できる。

## **C. 2度寝判定コード(疑似コード)**

state \= 'SLEEPING'  
wake\_time \= return\_start \= None

EXIT\_THRESH    \= 0.3   \# 体重の30%を下回ったら離床  
RETURN\_THRESH  \= 0.4   \# 体重の40%を超えたら再入床  
MONITOR\_SEC    \= 1800  \# 起床後30分監視  
CONFIRM\_SEC    \= 180   \# 3分継続で2度寝確定

while True:  
    weight \= get\_weight()   \# 正規化済み(体重比)  
    now \= time.time()

    if state \== 'SLEEPING' and weight \< EXIT\_THRESH:  
        wake\_time \= now  
        state \= 'AWAKE\_WINDOW'

    elif state \== 'AWAKE\_WINDOW':  
        if now \- wake\_time \> MONITOR\_SEC:  
            state \= 'DONE'  
        elif weight \> RETURN\_THRESH:  
            return\_start \= now  
            state \= 'RETURNED'

    elif state \== 'RETURNED':  
        if weight \< RETURN\_THRESH:  
            state \= 'AWAKE\_WINDOW'  
        elif now \- return\_start \> CONFIRM\_SEC:  
            trigger\_buzzer()  
            send\_notification()  
            state \= '2ND\_SLEEP'

# **付録E: 用語集**

本計画書で使用される技術用語の定義を記載する。

| 用語 | 意味 |
| ----- | ----- |
| **ロードセル** | 荷重を電気信号(微弱電圧)に変換するセンサー。本研究では50kg対応のハーフブリッジ型を使用 |
| **HX711** | ロードセル専用の24ビットADC(アナログ・デジタル変換)モジュール。微弱信号を増幅してデジタル化する |
| **ハーフブリッジ / フルブリッジ** | ロードセルの内部回路構成。ハーフブリッジ4個を組み合わせて1つのフルブリッジを構成すると総荷重と偏りが取得できる |
| **Raspberry Pi 4** | 英Raspberry Pi財団製の小型コンピュータ。GPIO経由でセンサーと接続でき、Pythonで制御できる |
| **GPIO** | General-Purpose Input/Output。Raspberry Piの汎用入出力ピン。デジタル信号の送受信に使う |
| **キャリブレーション** | センサーの生の値を実際の物理量(kg)に変換するための較正作業。ゼロ点とスケール係数を測定する |
| **状態遷移モデル** | システムの状態を定義し、条件に応じて状態を遷移させる設計手法。本研究ではSLEEPING〜2ND\_SLEEPの5状態 |
| **ルールベース** | 「もしXがY超えたらZ」のような固定条件で判定する手法。学習不要で軽量だが柔軟性は低い |
| **機械学習(ML)** | データから自動的にパターンを学習する技術。本研究ではロジスティック回帰・ランダムフォレスト等を使用 |
| **LLM** | Large Language Model(大規模言語モデル)。Gemma 4・GPT-4oなど。自然言語の生成・理解が可能 |
| **Ollama** | LLMをローカル環境で簡単に実行できるオープンソースツール。CPU/GPUどちらでも動作可能 |
| **特徴量** | 機械学習モデルへの入力データ。本研究では総荷重・荷重偏り・継続時間・時刻などを使用 |
| **F1スコア** | Precision(適合率)とRecall(再現率)の調和平均。0〜1の範囲で1に近いほど高精度 |
| **ツイストペア** | 2本の電線を撚り合わせた配線。差動ノイズを相殺できるためアナログ信号の長距離配線に有効 |
| **Webhook** | 特定のイベント発生時に外部URLへ自動的にHTTPリクエストを送る仕組み。Discord/LINE通知に使用 |
